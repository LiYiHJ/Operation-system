
# 运营系统设计：扩展性、高性能与多数据域接入优化设计报告

**版本**：2026-03-20  
**适用对象**：项目负责人 / 架构设计 / 后续接手开发者 / 前后端实现者  
**基线说明**：本报告基于你上传的所有 handoff / 设计稿 / 阶段报告，叠加你补充的“扩展性与性能分析”，并结合当前公开仓库分支 `checkpoint/import-stable-51f7df8` 的真实目录与关键代码结构整理而成。  
**报告目标**：在不降级、不减能力、不推翻当前基线的前提下，把系统升级为“面向未来 API + 多类导入 + 统一事实层 + 利润/策略/执行闭环”的可持续架构。

## 一、执行摘要

### 1. 总判断

当前项目不是“以后稍微一扩就会崩”的结构，相反，它已经具备几个很重要的长期资产：

- registry 驱动字段体系
- dual-gate / soft-gate / recovery / runtime audit 的可信导入框架
- 以主题事实层为目标的数据建模方向
- API + 导入并存的接入思路
- 前端主骨架与结构化导入页
- 利润、策略、执行闭环的方向稿

但按当前实现状态，它仍然属于 **“可扩展原型 / 中段工程”**，而不是 **“高成熟扩展平台”**。  
因此，下一阶段最正确的动作，不是继续堆页面或继续把更多规则塞进现有导入主文件，而是 **把扩展框架正式化**。

### 2. 本报告的核心优化结论

本次优化后的总方案，用一句话概括就是：

> **后端从“单一导入主文件 + 局部同步能力”升级为“数据集驱动的 ingestion 平台 + connector 框架 + 统一事实层 + 经营引擎层”；前端从“页面集合”升级为“Data Workspace + Domain Workspace + Action Workspace 三类工作台”。**

同时明确两条轴：

- **数据轴**：按 `dataset_kind` 管统一模型  
- **接入轴**：按 `import_profile / connector_profile` 管来源差异  
- **动作轴**：按 `suggestion -> approval -> push -> callback -> audit` 管执行闭环

这三条轴合起来，才能同时解决：

- 未来 API 越来越多
- 许多关键数据仍然只能靠导入
- 导入和 API 不能各走各的
- 利润 / 定价 / 评分 / 策略不能停留在孤立页面
- 当前实现存在的并行代际、内存 session、同步阻塞问题

## 三、对你新增分析的吸收结果：采纳、修正与升级

### 1. 直接采纳的判断

以下判断与现有报告方向一致，直接采纳：

- 当前项目 **方向正确，但成熟度尚未达到高性能平台**
- registry 驱动字段体系是长期最有价值的资产之一
- 当前导入模块不是简单上传，而是“可治理的数据入口”
- 事实层已经有主题化建模方向，不应回到大宽表
- API + 导入双路径是正确路线
- 并行代际代码、内存 session、同步 pandas 主流程，是当前扩展性与性能的主要风险点
- 导入未来必须 **profile 化**
- API 接入未来必须 **connector / adapter 化**

### 2. 需要修正的地方

你补充分析里强调“导入按 profile 化”，这点是对的，但还不够完整。  
本报告把它升级为 **双层建模**：

- `dataset_kind`：定义“这是什么业务数据”
- `import_profile` / `connector_profile`：定义“这份数据以什么形状和来源接入”

举例：

- `dataset_kind = ads`
  - `import_profile = ads_campaign_daily_excel`
  - `connector_profile = ozon_ads_campaign_daily_api`

- `dataset_kind = reviews`
  - `import_profile = shop_review_export_csv`
  - `connector_profile = marketplace_review_pull_api`

这样可以避免将来“按来源建模”，导致前端、事实层、利润引擎都被来源牵着走。

### 3. 升级后的总原则

本报告最终采用的原则不是“一个总导入器统吃全部”，也不是“每个数据域各自搞一套”，而是：

- **统一模型层**：用 `dataset_kind` 固定核心对象
- **统一批次层**：无论 API 还是文件，都用 batch / gate / audit 统一表达
- **统一动作层**：利润、定价、停投、补货、下架等，都用 action contract 管理
- **差异吸收层**：用 profile / connector / mapper 吸收来源差异

## 五、当前最核心的结构问题

### 1. 并行代际主线未彻底统一

当前仓库里同时存在：

- 顶层 `api.py` 的简化入口
- `api/app.py` 的模块化 Flask 主线
- 顶层 `import_service.py`
- `services/import_service.py`
- `ingestion.py`
- `intelligent_field_mapper_v3.py`
- 多份 `.bak_*` / `.disabled` / patch 脚本残留

这会带来两个严重问题：

1. 扩展时不知道新能力该接在哪条主线  
2. handoff、代码、前端契约之间容易漂移

### 2. 当前导入状态仍偏“进程内会话”

`services/import_service.py` 当前有明显的内存 `_sessions` 机制。  
这对于当前单机调试或小规模内部使用尚可，但不适合：

- 多用户
- 多 worker
- 大文件
- 长时导入
- 导入恢复
- API 同步与人工导入并行

### 3. 当前同步/导入仍偏同步长请求 + pandas 内存处理

这使得系统当前更像“内部运营系统 + 小中规模批处理”，而不是“高吞吐平台”。  
因此下一阶段的正确目标不是立刻微服务化，而是先把：

- batch 持久化
- job queue
- raw/staging/fact 闭环
- connector/profile 框架

做成稳固的模块化单体。

保存：

- 原始文件

- transport

- Order / OrderItem

- Refund / AfterSales
- InventorySnapshot
- PriceSnapshot
- StoreHealth

#### L6 - Engine / Strategy 层

- 评分影响模型
- 风险模型
- 产品分层

- 人工确认

## 十、导入与 API 的统一接入方案

### 1. 不再让文件导入和 API 接入各走各的

统一原则：

- 文件导入和 API 拉取都创建 batch
- 都写 raw
- 都进入 staging / normalize
- 都跑 gate / importability
- 都输出 audit
- 都按 dataset_kind 落 fact
- 都形成可追踪 run record

### 2. 统一批次 contract

建议统一字段：

- `batch_id`
- `source_type` (`file` / `api` / `manual`)
- `provider`
- `dataset_kind`
- `profile`
- `transport_status`
- `semantic_status`
- `importability_status`
- `gate_mode`
- `quarantine_count`
- `imported_rows`
- `mapping_summary`
- `audit_summary`
- `created_at`
- `confirmed_at`
- `replay_of_batch_id`
- `idempotency_key`

### 3. 为何这是升级而不是重做

因为当前导入主线已经有：
- parse
- mapping
- manual overrides
- confirm
- audit

而 integration/service 已经有 provider/config/scope/sync 雏形。  
这次只是把两者正式收成同一 batch 语义，而不是重写两套系统。

## 十一、性能与高扩展路线

### 1. 当前不建议直接上微服务

当前阶段正确路线是：

- 先做 **模块化单体**
- 再把 batch / queue / connector / registry / domain 边界做清
- 等平台、店铺、同步任务规模上来，再拆 task worker 与 connector worker

### 2. 第一轮必须做的性能性改造

#### P0 - 把 session 从进程内迁出
至少做到：

- session / parse result / confirm result / manual overrides / audit
- 落 DB
- 或 Redis + DB

#### P1 - 引入 job queue
先覆盖：

- API sync
- 大文件导入
- 回放
- 重试
- 推送任务

#### P2 - 把 raw/staging/fact 做成持久化闭环
尤其 API 同步不能长期停留在“拉完即 upsert 最小字段”的模式。

#### P3 - 做幂等与补偿
必须具备：

- idempotency key
- retry policy
- dead-letter
- replay
- partial failure compensation

### 3. pandas 的位置

pandas 不是必须立刻废掉。  
当前阶段正确定位是：

- parser / normalize / small-medium batch transform 的工具

而不是：

- 长期承担高并发同步执行器
- 通过 HTTP 长请求直接跑完大批量接入

也就是说：**保留其工具价值，去掉其系统主流程地位。**

## 十二、必要的“去运行面动作”，以及为什么这不是降级

### 1. `.bak_*`、`.disabled`、patch-heavy 脚本移出主运行面

不是删除，是归档。  
优点：

- 搜索结果干净
- 权威主线清楚
- 新接手者不会误判
- 历史证据仍在

### 2. `DataImport.tsx` 不再做第二主入口

不是删导入，是统一导入入口，减少状态分叉。

### 3. `DecisionShowcase.tsx` 转内部工具区

不是删决策相关能力，而是把“演示页面”和“正式动作链”分开。

### 4. `profit_registry_service.py.disabled` 退出运行面

不是放弃利润 registry，而是把其设计意图正式并入主线 `registry/profit_registry.py`。

## 十五、最终结论

本次优化设计不是否定你现有项目，而是把它从：

- 可扩展原型
- 中段工程
- 导入可信性强但扩展框架尚未正式化

升级成：

- 主线清晰的模块化单体
- 统一 batch / dataset / action contract
- 同时支持 API 与文件导入
- 面向利润 / 评分 / 策略 / 执行闭环的经营系统骨架

最重要的一句结论是：

> **未来更多 API 能接，更多 API 拿不到的数据继续靠导入也能接；但要做到真正易扩展、可长期维护、性能可演进，下一阶段必须从“功能增加”转向“主线统一 + profile/connector 框架化 + batch 持久化 + 统一事实层 + 工作台化前端”。**

这份方案的所有“去除”动作，本质都不是减能力，而是：

- 去掉并行歧义
- 去掉运行面噪音
- 去掉重复入口
- 去掉错误扩展方式

从而保住并放大你当前最有价值的资产。
