# 最终风险清单（P0 / P1 / P2）

## P0（阻塞“正式使用前确认”）
1. **主库业务基线为空**
   - 现象：核心事实表与闭环表当前 count 为 0。
   - 风险：页面/API 即使 200，也无法证明生产可用性。
   - 证据：`final_readiness_live_checks.json.mainDbCounts`。

2. **真实报表 confirm_import 稳定性不足**
   - 现象：A/B 真实报表 parse 可映射但 confirm failed（errorRows 高）。
   - 风险：导入看似成功、实际不可落库或部分失败，影响每日运营节奏。
   - 证据：`final_readiness_import_checks.json.importChecks[*]`。

3. **批量路径与失败恢复证据不足**
   - 现象：单链路可跑，但缺系统化批量失败恢复/回放门禁。
   - 风险：高峰期推送失败无法快速恢复。

## P1（上线前必须明确并至少一次演练）
1. **真实外部环境验收未完成（Ozon + 销售后台）**
   - 风险：mock 成功不代表真实链路可用。
   - 需补：真实凭证最小 scope 联通、真实回包校验、失败告警链路。

2. **权限与审计仍偏轻量**
   - 风险：关键动作权限边界不足，审计追责能力有限。
   - 需补：admin/operator/viewer 最小 RBAC 与审计事件。

3. **全站 UAT 回归缺少“有数据基线”**
   - 风险：空态回归掩盖真实业务问题。

## P2（不阻塞 UAT，但建议持续优化）
1. 启动流程统一脚本与环境自检（依赖告警收敛）
2. 导入错误报告可解释性（按字段/规则分组输出）
3. 运营看板空态体验优化（明确“无数据”原因与补救入口）
