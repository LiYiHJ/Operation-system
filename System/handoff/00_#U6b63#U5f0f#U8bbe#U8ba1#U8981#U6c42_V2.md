# 00_正式设计要求_V2

## 1. 文档定位

本文档是当前系统的正式实施要求，面向 Codex / 工程实现人员。  
本轮目标不是重写系统，不是做一个“能跑的轻量版后台”，而是：

**在保留现有前端完整产品形态与页面结构的前提下，高保真打通后端高质量能力，完成数据导入、入库、分析、利润、策略、决策与执行闭环。**

本轮工作的核心不是“多做新页面”，而是：

- 保留现有前端完整工作台
- 启用后端成熟模块
- 让数据真正进入数据库
- 让现有页面用真实数据运转
- 让利润与决策成为经营中枢
- 让批量推荐、整体预测、人工确认、后台推送形成闭环

---

## 2. 当前系统事实（必须基于真实现状实施）

### 2.1 前端现状

当前前端已经具备完整产品工作台形态，不是 demo，不允许被降级处理。

已知前端特征：

- 已有 10 个页面
- 已有大量图表、表格、卡片、按钮与复杂交互
- 已有 Dashboard、Import、Profit、ABC、Price、Funnel、Inventory、Ads、Strategy、Decision 等完整页面分工
- 页面整体风格已形成统一后台语言：顶部导航、店铺切换、左侧菜单、卡片化主内容区、表格与 badge 驱动的操作界面

结论：

> 前端不是“需要重新设计的壳”，而是“必须保留并强化的资产”。

### 2.2 后端现状

后端已有高价值能力，但未被充分接通和使用。

已知后端特征：

- 已有 Flask + Python 路由与服务结构
- 已有多个服务模块与数据库模型
- 已有 30 个左右数据表
- 已有 ImportService、IntelligentFieldMapper、ImportDiagnoser、ProfitService、ProfitSolver、StrategyEngine、WarRoomService 等可复用模块
- 但大量 analysis / ads / import / strategy 深层能力未完整接通
- 数据库大量表空转，使用率很低

结论：

> 当前最大问题不是“后端没写”，而是“后端写了但没接起来”。

### 2.3 当前最严重问题

1. 导入链路割裂
   - 前端做了大量解析
   - 后端 Import 三件套未真正接入
   - `/api/import/parse` 缺失或未真实完成
   - `/api/import/confirm` 未真实完成
   - 导入后未形成真实业务页面更新

2. 专题分析页大量为前端 mock
   - ABC
   - Price
   - Funnel
   - Inventory
   - Ads

3. 数据模型空转
   - 维度表、事实表、领域表大量未启用
   - 仅少数系统表被部分使用

4. 利润系统还停留在“计算器级”
   - 尚未成为多算法、多来源、可覆盖、可对比的经营中枢

5. 决策系统还停留在“建议列表级”
   - 尚未成为“先预演、再确认、再执行、再回写”的控制台

## 3. 本轮总目标

### 3.1 总体目标

在不删减现有前端页面、模块、复杂交互和界面设计的前提下，对系统做高保真实现与升级，使其真正成为：

**Ozon 跨境电商智能运营工作台 + 数据导入整合系统 + 利润与定价中枢 + 策略与决策控制台**

### 3.2 本轮必须实现的核心闭环

必须打通以下主链路：

**文件导入 → 后端智能处理 → 入库 → Dashboard / 分析 / Profit / Strategy / Decision 真数据驱动 → 用户确认执行 → 推送销售后台**

### 3.3 本轮必须达成的业务结果

1. 保留前端所有核心页面与模块  
2. 真实打通数据导入链路  
3. 启用后端 Import 三件套  
4. 让 Dashboard 具备“经营开工台”能力  
5. 让 Profit 升级为“算法利润工作台”  
6. 让 Price 升级为“促销调价驾驶舱”  
7. 让 Decision 升级为“组合决策预演台”  
8. 轻量接入登录与提醒，不破坏主界面  
9. 为新订单提醒、新评价提醒、订单处理、评价回复预留入口

## 4. 不可违背的硬约束

### 4.1 不允许删减前端功能

Codex 不得以任何“优化”“重构”“简化”为理由：

- 删除页面
- 合并页面后导致信息密度下降
- 删除图表、表格、步骤条、操作区
- 将多步骤流程压缩成单页面简单表单
- 将复杂分析页降级为普通 CRUD

### 4.2 不允许推翻现有前端产品结构

允许：

- 结构优化
- 组件拆分
- 状态管理优化
- 接口打通
- 真实数据接入
- 交互增强
- 可维护性提升

不允许：

- 重做成另一套产品
- 破坏现有视觉语言
- 把现有工作台做薄

### 4.3 不允许浪费成熟后端模块

以下后端模块优先复用与接线，不得轻易重写：

- ImportService
- IntelligentFieldMapper
- ImportDiagnoser
- ProfitService
- ProfitSolver
- StrategyTaskService
- StrategyEngine
- WarRoomService
- DashboardService

### 4.4 不允许继续让前端长期承担后端核心逻辑

以下必须以后端为主：

- Excel / CSV 复杂解析
- 表头检测
- 平台识别
- 字段智能映射
- 数据清洗
- 数据验证
- 去重
- 正式入库

前端保留：

- 上传交互
- 步骤流程
- 结果预览
- 人工修正
- 结果确认
- 状态反馈

### 4.5 不允许 mock 冒充完成

如果某页本轮仍无法接真实数据，必须明确标记待接入，不可伪装成已完成。

逐模块列出：

- 路由文件
- service 文件
- solver / engine / mapper / diagnoser 文件
- 数据库模型
- 已调用情况
- 未调用情况
- 空实现接口
- 返回 mock 的接口
- 空转数据表

### 5.3 审计输出必须回答的问题

- 哪些前端模块必须原样保留
- 哪些后端模块必须先接
- 哪些接口必须补
- 哪些表必须优先启用

## 7.4 促销调价驾驶舱（Price 升级为价格与促销中枢）

### 页面落点
`/analysis/price` 或现有 Price Strategy 页面

### 业务目标
解决以下真实问题：

- 平台促销有额外曝光，但平台折扣过大，很多 SKU 参加即亏
- 单纯提价参加活动，会导致日常价虚高，影响自然订单
- 平台允许商家自建促销，因此需要“平台活动 + 自建促销 + 日常售价”三层联动
- 各商品利润目标不同，必须批量管理，不可单 SKU 混乱维护

### 必须新增的 UI 区块

#### A. 三视图切换
- 日常售价视图
- 平台促销视图
- 自建促销视图

#### B. 策略组面板
按组管理 SKU，建议组：
- 引流组
- 标准利润组
- 高利润组
- 清仓组
- 大促参与组

每组配置：
- 目标净利率
- 平台活动最低利润线
- 自建促销默认折扣
- 是否允许活动前自动提价
- 是否允许自动转入自建促销

#### C. 主表格
必须包含：
- SKU
- 当前原价
- 当前成交价
- 目标净利率
- 平台活动折扣
- 平台活动净利率
- 建议提价后原价
- 自建促销折扣
- 自建促销净利率
- 推荐策略
- 风险等级
- 加入预演

#### D. 右侧解释板
显示：
- 推荐原因
- 平台活动为何不合适
- 自建促销是否可替代
- 保底利润线
- 调价后影响

### 必须支持的动作
- 生成推荐
- 按组批量加入动作池
- 标记观察
- 导出调价草案
- 进入预演台

### 必须接通的接口与数据
- 新增 `/api/analysis/price`
- SkuPriceMaster
- SkuCostMaster
- PriceChangeLog
- PromotionEvent
- ProfitAssumptionProfile
- FactProfitSnapshot

### 强制要求
此模块必须以“按组策略管理 + 批量推荐 + 先预演再执行”的形态实现，不能做成单 SKU 零散操作工具。
