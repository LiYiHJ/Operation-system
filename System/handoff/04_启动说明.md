# 04_启动说明

## 一、文档目的

本文档用于指导开发者或 Codex 在本地环境中启动当前 Ozon 智能运营系统，并完成基础联调验证。

本项目当前目标不是重新搭环境，而是在现有环境基础上继续高保真实现与升级。因此本说明优先面向：

- Windows 本地开发环境
- 前后端分开启动
- 数据库可本地运行
- 先跑通系统，再做导入链路与功能联调

---

## 二、启动前原则

1. 不要把真实密钥写进源码。
2. 使用 `.env` 或环境变量管理敏感信息。
3. 不要把 `node_modules`、`venv/.venv`、缓存目录一起打包给 Codex。
4. 如果当前本地环境已经能跑通，请优先复用现有环境，不要大改依赖版本。
5. 启动后优先验证：
   - Dashboard 是否可打开
   - Import 页面是否可进入
   - Profit 页面是否可进入
   - Strategy / Decision 页面是否可进入

四、环境准备
4.1 前端环境

推荐：

Node.js 18+

npm / pnpm / yarn 任选其一，但以项目现有 lock 文件为准

检查命令：

node -v
npm -v
4.2 后端环境

推荐：

Python 3.10+

pip

venv 或 .venv

检查命令：

python --version
pip --version
4.3 数据库环境

当前开发环境可允许：

SQLite（开发快速验证）

PostgreSQL（推荐联调和后续正式使用）

原则：

本地快速验证可以先用 SQLite

若已有 PostgreSQL 配置，则优先按现有配置启动

不要把实现锁死在 SQLite 特性上

五、环境变量说明

请不要使用真实密钥示例，统一使用占位符。

.env.example 可参考：

APP_ENV=development

FRONTEND_PORT=3000
BACKEND_PORT=5000

DATABASE_URL=sqlite:///./data/ecom_v51.db
# 或
# DATABASE_URL=postgresql://username:password@localhost:5432/ecom_v51

API_KEY=__REDACTED__
OZON_CLIENT_ID=__REDACTED__
OZON_API_KEY=__REDACTED__

SECRET_KEY=__REDACTED__

如果项目前后端分别使用环境文件，建议拆成：

frontend/.env.local

backend/.env

六、前端启动步骤

进入前端目录：

cd frontend

安装依赖：

npm install

或：

pnpm install

启动开发服务：

npm run dev

或：

npm start

启动成功后，记录：

本地端口

首页地址

是否有代理配置到后端

建议确认以下页面能进入：

/dashboard

/import

/profit

/strategy

/decision

七、后端启动步骤

进入后端目录：

cd backend

创建虚拟环境（如未创建）：

python -m venv .venv

激活环境（Windows PowerShell）：

.venv\Scripts\Activate.ps1

安装依赖：

pip install -r requirements.txt

如果项目使用 pyproject.toml 或其他方式，请按现有方式安装，不要强行切换。

启动后端服务（示例）：

python app.py

或：

flask run

或项目已有启动命令：

python -m src.ecom_v51.api.app

以你项目现有入口为准。

本轮必须确保数据库可创建基础表，并支持导入链路写入：

ImportBatch

8.2 初始化方式

如果项目已有迁移脚本，使用现有脚本。

确认数据库中已经创建关键表，并可连接查询。

九、启动后基础联调检查
9.1 前端页面检查

逐项确认：

Dashboard 可正常打开

Import 页面可正常打开

Profit 页面可正常打开

Strategy 页面可正常打开

Decision 页面可正常打开

9.2 后端接口检查

至少确认以下接口可响应：

/api/dashboard/overview

/api/dashboard/metrics

/api/import/parse（若本轮已实现）

/api/import/confirm（若本轮已实现）

/api/profit/solve

/api/strategy/list

9.3 前后端联调检查

至少确认：

前端请求能发到后端

后端跨域/代理配置正常

错误信息可见

页面不会因接口失败而整页白屏

十、推荐启动顺序

推荐顺序如下：

启动数据库

启动后端

启动前端

打开 Dashboard

打开 Import 页面

执行一次导入链路测试

再验证 Profit / Strategy / Decision 页面

原因：
导入链路是本轮最高优先级，一旦导入能真实入库，后续 Dashboard、分析、策略、决策页面才有真实数据可看。

十一、常见问题
11.1 前端能开，后端没数据

说明当前仍在读 mock 或接口未接通。
需要检查：

页面 API 是否真实调用后端

后端接口是否空实现

数据库是否实际写入

11.2 后端能跑，但导入后页面没变化

说明链路只跑到了“保存文件”或“解析成功”，没有真正入库或页面未读真实数据。
优先检查：

/api/import/confirm

ImportService 是否被实际调用

Fact / Dim / Snapshot 表是否写入

Dashboard / Analysis / Profit 是否读取真实表

11.3 页面报错但不知原因

要求前端统一处理：

loading

empty

error

retry

不能只显示控制台错误，必须有用户可读提示。

十二、启动完成的最低成功标准

满足以下条件，才算“系统启动成功并具备进入开发/联调状态”：

前端主页面可进入

后端服务可响应基础接口

数据库关键表已初始化

Profit 页面可进行基础计算

Strategy 页面可返回任务列表

Import 页面可进入并开始上传流程

至少具备执行导入链路测试的条件
