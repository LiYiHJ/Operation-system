# 导入模块与整系统阶段性交接分析报告

## 0. 文档目的

本报告用于把当前窗口内已经推进过的工作、已经形成的结论、尚未完成的目标、必须遵守的口径边界、以及下一位接手者应当如何继续执行，完整交接清楚。

这不是宣传文案，也不是概述摘要，而是**执行型 handoff 文档**。目标是让下一位接手者在不反复翻历史对话的情况下，能够准确理解：

1. 目前系统做到哪里了；
2. 哪些结论已经被证据支撑；
3. 哪些东西还不能宣布完成；
4. 下一步应该优先做什么；
5. 哪些历史坑已经踩过，不能再重复犯。

---

## 1. 项目整体背景

这是一个 **V5.1 跨境电商智能运营系统**，核心方向不是单一页面开发，而是一个围绕运营分析、导入、利润、策略决策、以及后续推送后台的完整运营系统。

系统范围至少覆盖：

- 运营总览 Dashboard
- 数据导入 Import
- ABC 分析
- 价格竞争力
- 转化漏斗
- 库存预警
- 广告管理
- 策略清单
- 后续利润计算与策略推送闭环

用户此前已经反复强调几个关键业务诉求：

1. **利润计算不能写死为单一算法**，必须支持可配置、可人工修正、可多算法共存；
2. 系统不能只“展示分析结果”，还要支持**批量决策、预测、确认后通过 API 推送销售后台**；
3. 当前阶段最先被重点推进的是 **导入模块**，尤其是：
   - xlsx / csv 真实样本验证
   - 双层门禁（dual-gate）
   - bad-header / 脏表头 / 多层表头的处理
   - 文案口径收敛，防止“证据不够但话说太满”

因此，这个系统并不是“页面差不多做完”就算完成；真正的核心是：

- **导入是否可信**
- **利润与策略是否能形成可执行闭环**
- **最终是否能用于真实运营决策**

3. “所有 bad-header / 脏表头场景都稳了”；

5. “导入模块已不存在结构/语义风险”；

### 4.3 当前最容易被误判的一点

**当前 bad-header 样本通过，并不 automatically 等于 recovery 机制已经有充分独立证据。**

因为最新样本结果里：

- preRecoveryStatus 已经是 passed；
- recoveryAttempted = false；

这说明当前这份最新证据体现的是：

- **去掉命名依赖后，基线语义终判直接通过**；

而不是：

- **恢复链路被触发后成功把样本从 risk 拉升到 passed**。

后者仍然需要独立样本证明。

## 8. 执行时必须遵守的要求

这一节是“硬约束”。

### 8.1 不要再写大结论

后续 PR、总结、交付说明，必须坚持：

- 证据到哪里，就写到哪里；
- 没有真实样本支撑的，不得写“已验证通过”；
- 没有泛化矩阵的，不得写“整体稳定”；
- 没有恢复触发证据的，不得把 recovery 写成“已充分验证”。

### 8.2 不要再回到文件名驱动逻辑

`file_name_risk_hint` 这类逻辑已经被清掉。

除非有非常明确的临时测试目的，否则不要再把：

- 文件名
- fixture 名称
- 测试样本别名

作为最终语义门禁的强制决策依据。

### 8.3 前后端字段必须同步

导入模块的 parse / confirm 字段已经变复杂，任何后续修改都必须同步：

- 后端返回
- `types/index.ts`
- DataImportV2 前端展示
- runtimeAudit
- 证据文档

否则很容易出现“后端字段改了，前端还按老口径展示”的情况。

### 8.4 证据文件必须跟真实运行结果一致

这次历史里已经出现过一类风险：

- 服务没起来，curl 失败；
- JSON 文件为空，json.load 报错；
- 但后面又继续往下跑总结。

后续必须严格避免这种证据污染。

要求：

1. 服务启动成功后再跑样本；
2. 生成的 JSON 先 `python -m json.tool` 检查；
3. 再写 markdown 总结；
4. 证据文档内容必须以最终 JSON 为准。

### 8.5 git 状态要清理

历史里多次出现：

- `System/data/ecom_v51.db`
- `System/src/uploads/*`

这些运行产物污染 `git status`。

后续执行时应：

- 不把 DB 变更误算为代码改动；
- 不把上传样本残留文件纳入提交；
- 证据文件和代码文件分开看。

因此下一步一定要补泛化样本。
