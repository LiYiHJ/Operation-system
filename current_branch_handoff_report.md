## 2. 当前阶段结论
- 中文样本的导入闭环已经打通：上传 -> 解析 -> 字段映射 -> 人工确认 -> confirm -> 缺失评分审计。
- 俄语 Ozon 报表已经可以进入映射页，但自动映射准确率仍不足，尤其体现在：说明文本被当作样本值、动态/占比/建议列不应直接映射到核心事实字段、`SKU` 与 `Артикул` 可能同时竞争 `sku`。
- 当前主目标不是“映射更多字段”，而是“保证核心事实层可信”。

## 3. 已完成事项
### 3.1 导入链路
- `DataImportV2` 已作为主导入页面使用。
- 上传后能够进入字段映射页。
- 中文样本可成功完成导入，缺失评分被按缺失事实保留。

### 3.2 后端稳定性
- 上传路由已恢复到可运行状态。
- `/api/reminders/list` 的时区比较问题已处理到不再持续干扰导入调试。

## 4. 当前核心问题
- 俄语报表中说明文本、动态列、占比列、建议列仍不能安全自动映射。
- 样本值历史上曾出现长说明文本、多列混采样等问题。
- 大量未纳入设计的字段需要后续进入候选池与扩展字段注册阶段。

## 5. 当前正确策略
### 5.1 Core-safe 模式
先保证核心字段可信：`sku`、`orders`、`order_amount`、`impressions_total`、`impressions_search_catalog`、`product_card_visits`、`add_to_cart_total`、`stock_total`、`rating_value`、`review_count`、`price_index_status`。

### 5.2 暂不自动映射的字段类别
- `Динамика`
- `Доля`
- `ABC-анализ`
- `Рекомендация`
- 长解释文本
- 时效/补货等尚未正式建模字段

## 6. 下一阶段建议
### 6.1 Phase B：映射准确率收口
- 清洗样本值来源，避免长说明文本进入 sample values。
- 收紧俄语规则，移除过宽关键词。
- 对核心目标字段做唯一性去重。

### 6.2 Phase C：扩展字段吸收
- 建立未映射字段候选池。
- 引入动态伴生字段模型。
- 按业务价值逐步把高频字段吸收进标准字段体系。

## 7. 与整系统主线的关系
当前导入模块方向与整系统主线一致：先做可信导入与事实层，再接利润引擎、评分诊断、策略建议与执行闭环；系统定位是跨境电商经营分析与运营中枢，而不是重履约 ERP 替代。
