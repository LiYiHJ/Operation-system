## 7. 这一轮为什么先做 Phase A，而不是直接提高导入成功率

这里是整个后续路线最容易被误解的地方。

### 7.1 Phase A 的目标不是“提高成功率”
Phase A 的目标是：

> **先把系统从“会误判成功”改成“会如实表达状态”。**

所以它要加的是：

- `importabilityStatus`
- `importabilityReasons`

它解决的是：

- 状态表达准确率
- 系统对外契约准确率
- API 集成时的安全性

而不是直接让 `cn_real_xlsx` 立刻导入成功。

### 7.2 为什么必须先做这一步
如果现在跳过 Phase A，直接去做 entity key / 值级识别 / 主键救援，会出现一个很严重的问题：

> 你根本分不清：
> - 是识别错了
> - 还是识别对了，但提交规则卡住了

而当前中文样本就是典型反例：

- semantic 已通过
- 但提交层全隔离

所以 Phase A 是“状态校准层”，不是“智能增强层”。

## 11. 这些设计对以后接 API 有什么影响

这是用户特别关心的点，不能漏。

### 11.1 `importabilityStatus` 和以后接 API 强相关
以后无论接：

- 平台报表拉取 API
- ERP / 仓库系统 API
- 销售后台推送 API
- 策略执行 API
- 定时同步任务

都不能再只依赖：

- `finalStatus`

而应该依赖：

- `transportStatus`
- `semanticStatus`
- `importabilityStatus`
- `importabilityReasons`
- `importedRows`
- `quarantineCount`

### 11.2 如果不拆层，会有什么风险
当前中文样本就是最好的反例：

- `finalStatus = passed`
- 但 `importedRows = 0`
- `quarantineCount = 77`

如果以后外部 API 只看 `finalStatus = passed`，就会误判：

- 数据已经进入系统
- 可以继续推送后台
- 可以继续触发策略 / 利润 / 库存动作

这是高风险行为。

### 11.3 为什么这也意味着“缺点”
因为一旦拆出 `importabilityStatus`：

- 前端要适配
- 回归脚本要适配
- 以后所有 API 调用方也要适配
- 不能再偷懒只看一个 `finalStatus`

这就是复杂度上升的代价。  
但这个代价是值得的，因为它换来真实可审计的状态契约。
